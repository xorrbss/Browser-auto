## #1 — checkbox/radio false-green (the headline correctness fix)

A bare `click` on a checkbox/radio TOGGLES, so if the page's initial state differs at replay the final
state is wrong yet the run passes green. Probes (agent-browser 0.27.0):
- `el.checked` in a capture-phase click listener IS the post-toggle state for a DIRECT click; for a
  LABEL click the recorded (label) emit sees PRE-toggle (control toggles after) — so `raw!==el` ⇒ flip.
- `find … check --exact` sets the state ABSOLUTELY (works). **`uncheck` is BROKEN** (success=false, box
  stays checked) — so an absolute uncheck is unavailable.

Design: capture.js click handler — a native `<input type=checkbox|radio>` (checkableType) that ends
CHECKED emits `check` (radio always; checkbox via the post/flip); a checkbox-UNCHECK stays a `click`
(uncheck broken — documented residual). build-flow maps `check` → `find … check`. Bug found by the
test and fixed: checkbox click also fires input+change, so capture recorded a spurious `fill` — the
input listener now skips checkableType. tests/capture-checkable.test.sh proves the records, the
build-flow mapping, AND that `check` is absolute (stays checked when the initial state differs = the
false-green a click would cause). Committed flows byte-identical (no check steps). README+SCHEMA updated.

## #3 — masking regression (PII guarantee)

The only existing masking test feeds a SYNTHETIC `masked:true` record and would pass even if
`sensitive()` returned false. tests/capture-masking.test.sh drives REAL sensitive fields through
capture.js — password (type), OTP (autocomplete), card (autocomplete), CVV (inputmode+name) — and
asserts each is `input_value:null, masked:true`, a benign email is NOT masked (value captured), and no
raw secret appears anywhere in the buffer.
