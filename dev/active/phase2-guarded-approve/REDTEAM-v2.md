## HIGH (1 confirmed-real — BLOCKER)

### PRESENCE-1 — the human-presence echo (`echoedDocNo`) is non-secret; the §11 CRIT-closer is illusory
- **Invariant:** I3, T5, I2 (and the §11 closure of v1-CRIT-1).
- **Attack:** v2 closes the v1 "nonce = CSRF token, not presence" CRITICAL partly via a human-presence factor: `/api/approve` must echo `echoedDocNo`, "a value the human read/typed from the LIVE rendered 문서번호" (DESIGN §5 L141-142, §2 L68/71/73; T5 L44). But on the 결재 path **문서번호 IS the doc_id** (recipes/SCHEMA.md:19; DESIGN §I4 L25 "live 문서번호 must equal the confirmed doc_id"). That doc_id is (a) already a field of the `/api/approve` body `{doc_id, nonce, echoedDocNo}`, (b) re-rendered in prepare's own response, and (c) handed to **every same-origin caller** before any approve: `queryApprovals` does `SELECT *` (lib/db.js:156) and `routes-rpa.js:37-38` returns those rows (doc_id + amount) to the client for both the `query` AND `approve` intents; `app.js:331/333` renders them. So a same-origin script reads `echoedDocNo` straight from data it already has and POSTs `/api/approve` with no human. The "stronger: re-type the amount" fallback (§5 L142, §13-Q3) is equally vacuous — amount is in the same row + render.
- **Does v2's guard close it?** **NO.** The echo provides zero presence entropy: it cannot distinguish a human from a programmatic same-origin caller, which is exactly what T5/§5/§11 claim it does. The ≤1-outstanding-nonce + refuse-enqueue-while-pending legs only SERIALIZE a script's loop (I2 "no >1 per human action" still violated — each iteration needs no human). T8/SameSite/CORS independently block the cross-origin browser case and the bare nonce already defeats blind CSRF, so the echo is redundant there too. **Not the §10/I7 residual:** §10 accepts only local code-exec as the OS user, but T5 explicitly claims the echo defends a "local/same-origin caller"; a same-origin script (CSP-bypassing XSS, malicious extension, tricked console/bookmarklet — see MODAL-1) is a distinct, lower-privilege class the design asserts the echo stops and the finding shows it does not. With the echo vacuous there is **no software human-presence factor at all** — per-item gating collapses to the operator's manual modal click, which a same-origin script bypasses by POSTing `/api/approve` directly.
- **Required revision (v3):** Drop the claim that `echoedDocNo` is a presence/security factor — it is at best a typo-guard, and for a no-auth loopback design no machine-readable echo of returned data can be one. Either (a) make the invariant honest: re-state I2/T5/§11 as "approve is SERIALIZED, not human-gated; there is no software presence factor; per-item gating rests on the operator's manual click + I7," and DROP the §11 "CRIT nonce=CSRF→…closed" claim; or (b) add a **true out-of-band per-item factor a same-origin reader cannot replay** — a server-issued challenge never returned in any HTTP body (e.g. an OS-credential / WebAuthn user-gesture prompt at `/api/approve`, made MANDATORY per item, not the §13-Q3 high-amount-only option) bound into the consent. Until one of these lands, v2 is unsafe to implement for an irreversible financial action.

## MEDIUM (4 confirmed-real — carry-forward, fix or accept)

### FP-SUMMARY-2 — for the flagship app the LLM summary is the sole *structured* monetary signal, and it is outside the fingerprint
- **Invariant:** T4, I1.
- **Attack (no local code-exec; adversarial DRAFTER):** T4 closes the v1 stale-amount break only if the LLM summary is "advisory, never the sole monetary signal." For Hiworks the deterministic 금액 field does not exist (recipes/hiworks.json has no amount column/detail field; lib/db.js:138-139 "amount … is often only inside raw_text"), so the structured amount is always null; the only structured number the operator sees is the model-generated summary, which `summarize.js:28-31` instructs the model to surface and which `lib/llm.js:7-9` labels UNTRUSTED. A drafter prompt-injects the on-prem summarizer via document body text so the summary reads "5만원" while the true body figure is "5,000만원." The fingerprint binds `body_digest` faithfully, so click-time re-verify MATCHES; consent was anchored on a system-generated advisory string the fingerprint deliberately excludes. The §10/§13 "typed amount confirmation" control is structurally INAPPLICABLE (no deterministic amount to re-type).
- **Does v2's guard close it?** **Partially — not fully.** The v1 *stale-content* critical IS closed (v2 shows the LIVE raw body 원문, §2 L66, which contains the true figure and is `body_digest`-bound), so T4's literal "never the SOLE monetary signal" survives (the raw body is also a signal); only the narrower "sole *structured* signal" is true. No single v2 guard fully closes it: the echo binds identity not amount, the fingerprint binds content not comprehension, and the typed-amount tier cannot apply. This is a sharpened variant of the accepted §10 "adversarial-but-truthful doc" residual, but here the SUMMARY actively lies — it requires both successful injection AND operator negligence against an on-screen truthful body. Medium, not high.
- **Required revision (v3):** For any app lacking a deterministic structured amount the human reads DIRECTLY, do not market "deterministic amount" as the closer (§11 L197 is half-true for Hiworks — closure rests on `body_digest` + live raw body). De-emphasize/collapse the advisory summary in the approve modal and make the raw-body + deterministic-amount panel the primary surface; require typed re-entry of a deterministic amount for high-value flows; if no deterministic amount exists, fail closed or state the closure is body_digest-only. (Already flagged open at §13-Q3.)

### AUTHORITY-1 — I3 "structural, not convention" is overstated for in-process future code
- **Invariant:** I3, T11.
- **Attack:** I3 (§0 L20-22) and §5 (L146-149) claim the leaf's consent-token requirement makes the model→approve boundary structural so "a future mis-wiring cannot yield a model/script-driven approve" and "the model can neither mint nor present this token." Verified: the webui is a SINGLE node process whose only process boundary is spawning bash via `child_process` (spawn.js); there is **zero crypto/signing infra today**, so a future signer is necessarily co-located with the model router (agent.js + routes-rpa.js:38, the model-reachable `approve` branch that today returns candidates only). Every authority-granting capability — nonce mint (prepare), nonce consume + echo check, consent signer — lives in that process and is reachable by a one-branch in-process change (extend routes-rpa.js:38 to call prepare()+approve() over the model-narrowed doc_ids). That future code satisfies the Origin gate, mints+consumes the nonce, synthesizes `echoedDocNo` from the rendered 문서번호 it already extracts (PRESENCE-1), and calls the signer; the leaf then clicks on a validly signed, doc+fingerprint-bound token.
- **Does v2's guard close it?** **NO for in-process future code.** The consent-token-in-the-leaf genuinely closes v1's LOW ("out-of-process stray spawn drives approve") — that is real improvement. But a server-signed token proves "this node process minted it," not "a human consumed it," so the thing between model output and a real approve is today's early `return` — convention, the exact property v1 flagged and v2 claims to have made structural. Not the §10 residual: this is a code-path/trust-domain accuracy gap, not a separate local-code-exec attacker. No present runtime authority path exists (the model has zero authority as specified), so this is defense-in-depth / claim-accuracy, hence medium not high.
- **Required revision (v3):** Re-scope §11/I3 wording to "leaf token blocks out-of-process spawns" (NOT "blocks future mis-wiring"). To actually earn "structural," require a human-only secret consumed at `/api/approve` and bound into the consent (the same OS-credential/WebAuthn factor PRESENCE-1 needs), with mint/sign behind a separate trust boundary from the read/NL router; add a test asserting no code path from `/api/agent` reaches prepare/approve/sign.

### WATCHDOG-WEDGED-1 — kill-path `record stop` routes through the SAME wedged daemon it exists to escape → audit restored, VIDEO still lost
- **Invariant:** T9, I5.
- **Attack:** The project shares ONE agent-browser daemon (jobs.js:1-8) which owns ffmpeg; `record stop` is the only video finalizer (cleanup.sh:16, preflight.sh) and is a CLI call serviced BY that daemon. The watchdog's named rationale (jobs.js:15-19) is precisely a wedged/unresponsive daemon, so T9's bounded `agent-browser --session <S> record stop` routes through the wedged daemon, hits its bound, and returns without finalizing the `.webm`. `reapDaemon` (jobs.js:32-43, `daemon stop`) is likewise daemon-routed; `killTree(job.pid)` (jobs.js:137) kills only the bash→CLI job tree, not the separately-spawned daemon/ffmpeg. The interrupted-audit row is a daemon-independent sqlite write and survives; the video does not.
- **Does v2's guard close it?** **Audit half CLOSED, video half STILL-OPEN.** Not the §10/I7 residual — a wedged daemon is a documented infrastructural failure mode (the `wait --url` glob / os-error-10060 hang, Chrome/daemon hangs), needs no attacker and no code-exec. No safety/correctness impact (no double-approve, no audit↔status divergence; the interrupted row + synchronous `detail_after` snapshot survive as primary evidence). But **§11 OVERCLAIMS** it ("HIGH kill-path loses audit+video → T9" as closed) while I5 lists "video for every attempt" as non-negotiable for an irreversible action.
- **Required revision (v3):** Either record approve videos via a DEDICATED daemon/session stoppable independently of the wedged one, or finalize via direct ffmpeg signaling (SIGINT to flush the moov atom) rather than `agent-browser record stop`; and downgrade I5/§11 to "durable AUDIT guaranteed; video best-effort on a wedged-daemon/hard-kill path."

### IDOPEN-5 — the full I4 re-verify cannot run in-batch with the click → a real (narrow) TOCTOU window
- **Invariant:** T12, I4.
- **Attack:** T12 (L51) / §2 step 3 (L85) close the verify→click TOCTOU via "re-assert identity in the SAME batch as the click." But `BATCH` runs only agent-browser sub-commands (lib/env.sh:67-86 — no node, no sha256, no compare-to-stored). The load-bearing I4 re-verify (extract-detail node parse → exactly-one idLabel + title-equality + recompute `sha256(문서번호|제목|amount|body_digest)` → compare to the consent fingerprint) is therefore unavoidably a host-side round-trip OUTSIDE any batch; only a WEAK in-batch presence gate is batchable. Between the fingerprint snapshot and the click the page can change.
- **Does v2's guard close it?** **Partially.** Doc-substitution (advance to a different doc) is caught twice — the in-batch presence re-confirm bails when the confirmed 문서번호 is absent, and I6/step-5 re-opens by the original doc_id and finds no 완료 → ABORT. The surviving flavor is **same-doc content-substitution** (e.g. amount mutates while doc_no stays present) which passes the presence gate and is NOT re-fingerprinted at step 5. It requires an app that client-side live-mutates the loaded DOM without navigation, editable-while-pending docs, and winning a blind sub-second race on a serial path — uncommon, no code-exec — hence medium not high. Real gaps: "disable auto-advance" (L51) has **no concrete mechanism** and is absent from the §12 capture checklist (L211-215); there is no get-url-delta abort across the click.
- **Required revision (v3):** Acknowledge the full re-verify cannot be in-batch. Specify a concrete auto-advance disable (never use the app's next/auto-advance control — always navigate back and re-open the specific doc), add it to the §12 capture list, capture `get url` immediately before/after the click and ABORT on any un-initiated URL change, and keep the positive re-open-by-doc_id 완료 transition (I6) as the binding authority so a TOCTOU swap cannot be recorded as success.

## LOW (8 confirmed-real — carry-forward)

- **IDOPEN-1 — "count==1 --exact" overspecified for semantic targets.** True that 0.27.0 `get count` is CSS-only and `find … --exact click` silently clicks first-of-N (probe-record.sh:23-24, verify-flow.sh:148-151, assert.sh:108-114). But doc-open's preferred path is the doc_id urlTemplate gated by `wait_url` (§3 L100), exactly-one is checkable via node aria-parse (lib/aria.js:37-39 returns all hits; callers refuse 0/>1), and the real click-time authority is the positive 완료 re-verify (I6). **Fix:** state in §3/recipe that semantic `count==1` is a node snapshot-parse assertion (never `get count`), advisory pre-click, with I6 as the binding gate.
- **IDOPEN-4 — native `confirm()` has no 0.27.0 accept primitive; the recipe `confirm` block conflates a DOM button with a native dialog.** `nativeDialog:false` (a DOM button) is implementable; the unspecified `nativeDialog:true` path is not (footgun list: native-dialog invisibility; `--confirm-actions` is an operator gate, not a page-dialog acceptor). Fail-closed contains the blast radius. **Fix:** split the schema into `confirmKind = none|dom|native`; for `native`, require §12 to empirically prove an accept primitive exists BEFORE any build, else refuse (fail-closed) — never ship an unmatchable leg.
- **FP-BODYDIGEST-3 — `body_digest` is under/over-inclusive.** `bodyFromHeading()` runs first-heading→END (extract-detail.js:112), dedups via `prev.includes` (117), slices to MAX_BODY=8000 (24,122): tail past 8000 chars is unshown+unbound, and the to-END span includes the volatile 결재선/timestamps → fail-safe flapping. The `amount` leg scans the whole snapshot uncapped, so the most dangerous field is bound regardless; failures are fail-safe (abort). **Fix:** pin `body_digest` to an explicit recipe-declared content sub-region (body-heading → decision-line marker), forbid silent truncation (hash the full consented body or fail closed over a captured bound), add to §12, log flaps distinctly.
- **SUCCESS-4 — step-5 re-open has no cache-bust / server-fresh guarantee.** I6 promises a transition reflecting *committed* server state, but step 5 (§2 L87-88) re-opens + asserts the marker with no freshness gate; an optimistic-UI persisted to local storage could re-hydrate a "결재완료" DOM over failed-commit server data. (bfcache N/A — `AB_AUTH open` is a fresh goto; SW/304 caches only store server responses → fail-NEGATIVE.) Harm is fail-toward-NOT-approved (audit/operability), not unconsented approval. **Fix:** add "step-5 re-open must be a hard server-fresh fetch (cache-bust) and/or cross-check the doc left the operator's 미결 inbox on a fresh fetch" to §12; tighten I6 wording from "committed" to an enforced freshness/inbox-departure cross-check; fail closed if unprovable per app.
- **SHUTDOWN-EXIT-1 — server-shutdown grace is inert on Windows for hard kills.** shutdown() is synchronous `process.exit(0)` with no await and registered only for SIGINT/SIGTERM (server.js:482-491); taskkill /F, End Task, console-close (SIGHUP), reboot deliver no catchable signal, so §6/T9's "bounded grace + interrupted audit + record stop before reaping" mostly does not run. AUDIT durability survives via the fsync'd pre-click row + reconciliation; only in-flight VIDEO may truncate. **Fix:** make shutdown() async (await bounded record-stop + interrupted audit for SIGINT), add SIGHUP/console-control handling for the catchable subset, and document that taskkill /F video is best-effort (audit durable).
- **AUDIT-STATUS-DIVERGE-1 — failed/interrupted-but-status='approving' is not covered by the reconciliation predicate.** §2 step 6 (L90) writes a TERMINAL failed/interrupted audit while leaving status='approving'; §4 reconciliation (L131-133) only re-resolves "approving with ONLY a non-terminal audit." Such a doc matches neither path → permanently stuck 'approving' — a standing audit↔status divergence contradicting I5's "cannot diverge" (the kill path itself produces exactly this). Fail-closed (never re-offered; prepare/approve require strictly 'fetched'), so no double-approve; harm is orphaned non-retryable docs. **Fix:** on a terminal failed/interrupted result, atomically set `status='approve_failed'` in the SAME txn as the audit (mirror the success path); add a reconciliation sweep for `approving + terminal-audit`; reserve `approving + non-terminal` strictly for crash/kill gaps.
- **CONSENT-KEY-1 — consent-token signing-key handling is unspecified; the naive HMAC impl leaks the key via blanket env.** §5 says the bash leaf validates a "server-signed" token but never pins HMAC vs asymmetric or where the key lives; spawn.js:23 propagates the full `process.env` to every child (and agent.js:30 already loads config into env), so an HMAC secret in env reaches unrelated jobs and degrades I3 to convention. The key-theft/argv-replay *exploit* is the §10/I7 residual; the spec gap is not. **Fix:** specify an in-memory asymmetric keypair minted at server start (private key signs; leaf gets only the public key — env-exposing a public key is fine); never place a signing secret in `process.env`; pass the token via a 0600 temp file / stdin, not argv (consistency with T10); document that single-use is DB-enforced ('fetched'→'approving'), token only as integrity binding.
- **MODAL-1 — `util.el()` turns a document-derived `on*` string prop into an inline handler; the strict CSP is design-only with no enforcing test.** The textContent-for-children guard genuinely holds (el() uses createTextNode; zero `innerHTML` in webui), but util.js:13-14 routes an `on*` prop whose value is a STRING to `setAttribute`, producing an inline handler from document content; while a prepare nonce is outstanding the page JS holds nonce+doc_id, so any script execution can same-origin-POST `/api/approve`. Vector is poisoned remote doc content (a drafter), not code-exec. CSP (T13 L52) + the §9 inertness test are promised but unbuilt (true of all of Phase 2). **Fix:** harden el() to REJECT non-function `on*` values (throw/ignore, never setAttribute); emit a real strict `script-src 'self'` (no `unsafe-inline`) header for `/` and assets; extend the §9 test to assert an `on*`-string does not become a handler AND that the CSP header is present; prefer not holding the live nonce in page JS.
