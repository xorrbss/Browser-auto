# NL-driven RPA productization design

> **Status:** DESIGN TRACK / BACKEND CONTRACT IN PROGRESS, 2026-06-08. This document turns the
> current safe-but-limited natural-language command box into a product-grade RPA command center.
> It does not weaken the existing invariant: the model never drives the browser and never invents
> replay steps. The CommandPlan UI shell exists, but the durable plan endpoints below are the
> intended backend contract unless explicitly marked as the current bridge.

## 0. Current baseline

The repository is already strong for an internal, on-prem RPA console:

- `webui/agent.js` classifies Korean commands into a strict allowlist.
- `webui/routes-rpa.js` routes only deterministic actions: sync, summarize/enrich, query, review prepare.
- `webui/jobs.js` serializes browser jobs, which matches the one shared agent-browser daemon.
- `lib/db.js` has the generic `systems` and `records` store.
- `recipes/<app>.json` is the site-specific structure, not per-site code.
- `approve/approve-run.mjs` proves the effectful pattern: dry-run default, fail-closed guards, positive completion, audit, cap, kill-switch.
- `approve/flow-runner.mjs` exists as the standalone Playwright executor for future recorded actions.

The product gap is not browser automation. The gap is orchestration:

1. Natural language currently jumps from classify to route. There is no first-class plan object.
2. Query/sync/enrich are reasonably generic, but effectful action selection is still mostly approve-shaped.
3. The web UI shows logs/results, but not a durable command timeline with plan, dry-run, confirmation, and outcome.
4. Permissions are localhost/single-user plus a strong approve gate. That is acceptable for local ops, but not enough for a multi-user product.
5. Web UI routes and UX need product-level tests, not only driver/unit tests.

## 1. Product principle

**Natural language proposes a deterministic plan; it never executes live by itself.**

The model may:

- classify the command,
- choose a registered system,
- choose one already-enabled action from `recipe.actions`,
- propose a filter over DB-backed records,
- explain why it needs clarification.

The model must not:

- generate click steps,
- choose an arbitrary URL to drive,
- bypass dry-run,
- approve, submit, reject, pay, or otherwise commit an effectful action,
- narrow a reviewed approval list in a way that hides candidates without saying so.

Runtime remains:

```text
natural language -> validated CommandPlan -> deterministic dry-run/preview -> human confirmation -> queued deterministic driver -> durable result
```

## 2. CommandPlan

Introduce a durable plan object before any job is enqueued. It is a server-validated object, not a trusted model reply.

```jsonc
{
  "id": "cmd_20260608_001",
  "actor": "local-operator",
  "sourceText": "미결 지출 10건 요약하고 결재 준비해줘",
  "createdAt": "2026-06-08T08:00:00.000Z",
  "status": "planned",
  "intent": "prepare_action",
  "system": "hiworks",
  "action": "approve",
  "riskClass": "irreversible",
  "mode": "reviewed",
  "filter": { "keyword": "지출", "limit": 10 },
  "targetQuery": {
    "source": "records",
    "system": "hiworks",
    "where": { "keyword": "지출" },
    "limit": 10
  },
  "targets": [
    { "key": "IB-...", "title": "출장비 정산", "summary": "..." }
  ],
  "requirements": {
    "dryRun": true,
    "humanConfirm": true,
    "permission": "actions.approve.live",
    "sessionGate": true
  },
  "preview": {
    "message": "10건을 체크 가능한 검토 화면으로 준비합니다. 실제 결재는 아직 실행하지 않습니다.",
    "visibleFields": ["title", "drafter", "submitted_at", "summary"]
  },
  "refusal": null
}
```

Hard rules:

- A plan with `riskClass != "read"` cannot directly transition to live execution.
- A plan with unresolved targets is not executable.
- A plan with an unavailable or disabled `recipe.actions.<action>` is a refusal.
- A plan whose target count exceeds the action cap is a refusal or requires the operator to narrow it.
- A plan generated by NL is always revalidated from DB and recipe state immediately before dry-run and again before live.

## 3. State machine

All command execution should flow through this state machine:

```text
draft
  -> planned
  -> dry_running
  -> dry_failed | awaiting_confirmation
  -> queued
  -> running
  -> succeeded | failed | skipped | needs_review | cancelled
```

For read-only actions:

```text
planned -> running -> succeeded|failed
```

For effectful actions:

```text
planned -> dry_running -> awaiting_confirmation -> queued -> running -> terminal
```

Invalid transitions fail closed. For example, `planned -> queued` is rejected for `irreversible` actions.

## 4. Action registry

The canonical action catalog is `recipe.actions.<name>`, backed by the system registry.

Each action should expose product metadata, not only driver locators:

```jsonc
{
  "actions": {
    "approve": {
      "displayName": "결재",
      "description": "선택한 미결 문서를 승인합니다.",
      "class": "irreversible",
      "enabled": true,
      "reviewedOnly": true,
      "dryRunDefault": true,
      "permission": "actions.approve.live",
      "target": {
        "keyField": "doc_id",
        "titleField": "title",
        "summaryField": "summary"
      },
      "preview": {
        "fields": ["title", "drafter", "dept", "submitted_at", "summary"],
        "warnIfMissing": ["summary"]
      },
      "limits": {
        "maxTargetsDefault": 20,
        "maxTargetsHard": 100
      },
      "completion": { "kind": "marker-and-departure" },
      "capture": { "date": "2026-06-08", "capturedBy": "operator" }
    }
  }
}
```

The plan builder can only select actions from this registry. Unknown actions become `clarify` or `refusal`,
never fallback behavior.

## 5. Backend API contract

Implementation status as of 2026-06-09 release-candidate work:

- Current compatibility route: `POST /api/agent` still classifies and may enqueue legacy read-only
  sync/summarize jobs immediately. It does not execute approval.
- Durable CommandPlan routes are implemented: `POST /api/agent/plan`, singular/plural
  `GET /api/agent/plan(s)/:id`, `POST /targets`, `POST /dry-run`, `POST /confirm`, `GET /events`,
  and `GET /result`.
- Durable storage is implemented in `lib/db.js` as `command_plans` and `command_events`.
- The Command Center UI is connected to the CommandPlan API. It does not create a persisted plan on page
  load; the operator explicitly creates a plan, reviews targets, runs dry-run, and then confirms.
- Job result surface is implemented as `GET /api/jobs/:id/result`; SSE/logs remain observability.
- Action/status surfaces are implemented as `GET /api/actions`, `GET /api/systems/:name/state`, and
  `GET /api/systems/:name/actions`. The exact `/capabilities` alias/schema below remains a future
  compatibility improvement.

Keep the existing `/api/agent` as compatibility, but add explicit plan endpoints:

| endpoint | contract |
|----------|----------|
| `POST /api/agent/plan` | Body `{text, system?, selectedTargets?}`. Returns `{plan, capabilities, warnings}` or `{clarification}` / `{refusal}`. It persists a `CommandPlan` and **does not enqueue a job**. |
| `GET /api/agent/plans/:id` | Returns the persisted plan, current status, latest dry-run/live result summary, and plan hash after refresh/restart. |
| `POST /api/agent/plans/:id/dry-run` | Body `{planHash}`. Revalidates the plan, target records, recipe/action capability, and caps. For effectful actions it enqueues a deterministic dry-run; for read-only plans it performs/returns the read preview. |
| `POST /api/agent/plans/:id/confirm` | Body `{planHash}` plus any explicit confirmation fields. Only valid after a passing dry-run for effectful actions. Requires session, present same-origin Origin/Referer, permission, unchanged plan hash, fresh capability status, and revalidated targets. |
| `GET /api/agent/plans/:id/events` | Returns durable command events; `?stream=1` may stream SSE using the same event records. |
| `GET /api/agent/plans/:id/result` | Implemented minimal durable shape `{plan,dryRun,confirmation}`. Per-target irreversible evidence stays in `data/approve-audit.jsonl` plus `GET /api/jobs/:id/result`; a normalized audit-linked terminal shape is still future work. |
| `POST /api/agent/plans/:id/cancel` | Future work. Queued job cancellation exists at `/api/jobs/:id/cancel`; live approve recovery uses the leaf kill-switch/STOP semantics, not rollback. |
| `GET /api/systems/:name/capabilities` | Future alias/schema. Current implemented surfaces are `GET /api/actions`, `GET /api/systems/:name/state`, and `GET /api/systems/:name/actions`, including disabled reasons. |

The existing `POST /api/approve/run` can remain as the low-level leaf route. The command layer should call it
only after plan confirmation. `/api/agent` must not call it directly.

### Dry-run and confirm gates

Read-only plans (`query`, read preview, record lookup) can run without confirmation. Browser-driving but
read-only plans (`sync`, `enrich`) still enqueue through the serial queue because they share the
agent-browser daemon, but they do not need a live confirmation gate.

Effectful plans (`riskClass == "reversible"` or `"irreversible"`) must satisfy every gate:

- `dry-run` is required and defaults to no live click. It must use the same deterministic leaf/driver and
  stop before the irreversible step.
- `confirm` is rejected unless the latest dry-run passed for the same `planHash`, target-set hash, action
  version, and recipe hash. The caller must echo the exact `planHash`, `targetSetHash`, and `dryRunHash`.
- `confirm` re-reads the DB records and recipe immediately before queuing live work. Stale title binding,
  missing target records, disabled action metadata, or changed capability status moves the plan to
  `needs_review` or `refused`.
- `confirm` requires the existing web session cookie and present same-origin Origin/Referer gate for
  effectful routes. In local mode the actor is `local-operator`; future multi-user mode maps the same
  permission string onto real users.
- A changed reviewed target set invalidates the previous dry-run. A confirmed plan cannot be confirmed
  again; live execution needs a new plan/dry-run.
- The model has no transition into `confirm`. It can only produce a plan that a deterministic validator
  accepts or refuses.

### Job result endpoint

`GET /api/agent/plans/:id/result` should normalize job output into a stable product shape:

```jsonc
{
  "planId": "cmd_20260608_001",
  "status": "succeeded",
  "job": { "id": "j42", "status": "done", "exitCode": 0 },
  "dryRun": { "status": "passed", "at": "2026-06-08T08:10:00.000Z" },
  "live": { "status": "succeeded", "at": "2026-06-08T08:12:00.000Z" },
  "targets": [
    { "key": "IB-...", "status": "approved", "reason": "confirmed marker + departure" }
  ],
  "audit": [
    { "source": "approve-audit", "doc_id": "IB-...", "stage": "confirmed" }
  ],
  "error": null
}
```

For approve-like leaves, this endpoint links to `data/approve-audit.jsonl` as the irreversible source of
truth. The command result may cache a parsed summary, but it must not replace the append-only audit.

### System and action capability status

Capabilities are server-derived facts, not model output. The intended response for
`GET /api/systems/:name/capabilities`:

```jsonc
{
  "system": {
    "name": "hiworks",
    "registered": true,
    "recordCount": 177,
    "recipe": { "present": true, "valid": true, "hash": "..." },
    "auth": { "agentBrowser": true, "playwright": true },
    "targetUrl": true
  },
  "actions": [
    {
      "name": "sync",
      "riskClass": "read",
      "state": "enabled",
      "requirements": ["registered", "targetUrl", "agentBrowserAuth"]
    },
    {
      "name": "enrich",
      "riskClass": "read",
      "state": "enabled",
      "requirements": ["recipe.detail", "agentBrowserAuth"],
      "externalDependencies": ["SUMMARY_MODEL optional for summaries"]
    },
    {
      "name": "approve",
      "riskClass": "irreversible",
      "state": "enabled",
      "requirements": ["recipe.actions.approve.enabled", "playwrightAuth", "listUrl", "dryRun", "humanConfirm"]
    },
    {
      "name": "reject",
      "riskClass": "irreversible",
      "state": "needs_implementation",
      "requirements": ["recorded action capture", "dry-run", "live disposable verification"]
    }
  ]
}
```

The plan builder can only select actions with `state == "enabled"` unless it returns a clarification or
refusal explaining what is missing.

## 6. Durable event log

Command-level observability is implemented in `lib/db.js`:

```sql
CREATE TABLE command_plans (
  id           TEXT PRIMARY KEY,
  actor        TEXT,
  source_text  TEXT,
  plan_json    TEXT NOT NULL,
  status       TEXT NOT NULL,
  risk_class   TEXT NOT NULL,
  system       TEXT,
  action       TEXT,
  job_id       TEXT,
  created_at   TEXT NOT NULL,
  updated_at   TEXT NOT NULL
);

CREATE TABLE command_events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  plan_id    TEXT NOT NULL,
  at          TEXT NOT NULL,
  stage       TEXT NOT NULL,
  detail      TEXT,
  data_json   TEXT
);
```

For approve-like actions, keep `data/approve-audit.jsonl` as the irreversible leaf source of truth. The
command event log links to it instead of replacing it.

## 7. Permissions

Local single-user remains the default deployment, but the design should not bake that in.

Minimum role model:

- `viewer`: read runs, records, summaries, audit.
- `operator`: register systems, auth, analyze, sync, enrich, create plans, dry-run.
- `approver`: confirm reviewed irreversible actions.
- `admin`: manage recipes, action enablement, model endpoint, retention.

Implementation path:

1. For local mode, use a single implicit `local-operator` actor, but record it in every plan and event.
2. Require the existing session cookie + same-origin gate for every effectful confirmation route.
3. Later, add real users and map them to the same permission strings. Do not change leaf drivers for auth.

## 8. Web UI

Replace the single command output box with a command center:

- Command input.
- Plan card: interpreted intent, system, action, risk, target count.
- Target preview table/cards with the exact fields the action will bind to.
- Dry-run result section with per-target status.
- Confirmation controls:
  - read-only: no confirmation.
  - reversible: confirm button.
  - irreversible: checkbox selection, dry-run required, explicit live confirmation.
- Event timeline with links to job logs and action audit.

The current approvals and systems views already have most pieces. This design mainly makes them first-class
and durable instead of transient UI state.

## 9. Tests

Add tests before broadening live behavior:

- `tests/agent-plan-unit.test.sh`
  - model JSON is untrusted, unknown actions refused, limits clamped, disabled actions refused.
- `tests/agent-plan-state-unit.test.sh`
  - illegal transitions fail closed.
- `tests/action-registry-unit.test.sh`
  - recipe action metadata validation.
- `tests/command-events-unit.test.sh`
  - command/event persistence, rollback, link to job id.
- Web UI Playwright smoke tests:
  - plan creation renders a preview,
  - dry-run streams and reaches terminal state,
  - live confirm is unavailable until dry-run passes,
  - `approve` intent from NL never calls `/api/approve/run` directly.

The existing `tests/agent-isolation-unit.test.sh` remains load-bearing and should be extended, not removed.

## 10. Phased implementation

### Phase 1: durable plans for current intents

- Add `CommandPlan` builder for `sync`, `summarize/enrich`, `query`, `review`.
- Add DB tables and event log.
- Keep using existing drivers and routes.
- UI shows plan + event timeline.

### Phase 2: action registry bridge

- Extend recipes with product metadata for `actions.approve`.
- Let NL select only enabled actions.
- Convert reviewed approve preparation into a plan.
- Low-level `/api/approve/run` remains unchanged.

### Phase 3: effectful confirmation gate

- Add `/api/agent/plans/:id/dry-run` and `/confirm`.
- Confirm calls the existing low-level route only after dry-run and permission checks.
- Persist plan hash and revalidate target/title/action immediately before confirm.

### Phase 4: recorded non-approval action

- Wire `approve/flow-runner.mjs` into a new action leaf for one disposable, non-approval action.
- Require per-action capture, dry-run, live-on-disposable, and sign-off before `enabled:true`.

### Phase 5: multi-user hardening

- Real auth, roles, per-user audit, retention controls, model endpoint governance.
- Do not expose local process-spawning routes without this.

## 11. Explicit non-goals

- No open-ended live LLM browser agent.
- No model-authored click sequence.
- No live irreversible execution from a single natural-language message.
- No unattended live approve through the scheduler.
- No generic cross-system write orchestration until single-system actions have durable plan semantics.

## 12. Remaining external dependencies

These are not solved by the CommandPlan contract itself:

- Private/TLS on-prem model endpoint for classification and summaries. The current local model guard warns
  on public plain HTTP; production should set `LLM_REQUIRE_PRIVATE=1` after the endpoint is private/TLS.
- Durable command storage is implemented as `command_plans` and `command_events`. The remaining product
  improvement is a richer normalized terminal result that joins command state, job result, and
  `data/approve-audit.jsonl`.
- Consolidated capability backend exists through `/api/actions`, `/api/systems/:name/state`, and
  `/api/systems/:name/actions`. The exact `/api/systems/:name/capabilities` alias/schema is future work.
- Playwright approve state per app (`approve/<app>.pw-state.json`) and a configured list URL remain required
  for approve-like actions.
- `fixtures/auth/<app>.state.json`, registered recipes, and `recipe.detail` remain required for sync/enrich.
- Real multi-user auth/roles are still future work; local mode records `local-operator`.
- Product-level route and UI smoke tests are part of the RC gate before broadening live behavior.

## 13. Acceptance criteria for this track

- A Korean command produces a visible, durable plan before work starts.
- The plan can be inspected and refused without side effects.
- Read-only plans can run immediately.
- Effectful plans require dry-run and explicit confirmation.
- Every terminal result is visible in the web UI after refresh/restart.
- The full `bash run.sh` gate remains green.
